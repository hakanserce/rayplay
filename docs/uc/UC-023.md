# UC-023: Codec Fallback and Next-Generation Codec Support

- **UC ID:** UC-023
- **Title:** Codec Fallback and Next-Generation Codec Support
- **Priority:** P2
- **Estimated Complexity:** M

## Description

As a user, I want RayPlay to fall back to an alternative codec when HEVC NVENC is
not available on the host, and to support newer hardware-accelerated codecs like AV1
when both sides support them, so that the best possible codec is always used.

## Acceptance Criteria

1. If HEVC NVENC is unavailable on the host (e.g., older GPU), the system automatically
   falls back to H.264 NVENC and notifies the user
2. AV1 encoding via NVENC (RTX 4000 series and later) is supported as an opt-in
   higher-quality option
3. Codec selection is negotiated during session handshake based on host encoder
   capabilities and client decoder capabilities
4. The user can override automatic codec selection via CLI option
5. All supported codecs meet the <5 ms encoding latency target on compatible hardware

## Technical Approach

Extend the `VideoEncoder` trait in `rayplay-video` to enumerate available hardware
encoders at startup. Session negotiation ([UC-015](UC-015.md)) exchanges the encoder list and selects
the best mutually supported codec in priority order: AV1 > HEVC > H.264. The encoding
and decoding stages already abstract over codec via the trait; adding a new codec means
implementing the trait for the new NVENC preset and the corresponding decoder on the
client side.

## Dependencies

- [UC-015](UC-015.md) (Session Resilience)
