# UC-015: Session Resilience

- **UC ID:** UC-015
- **Title:** Session Resilience
- **Priority:** P1
- **Estimated Complexity:** M

## Description

As a user, I want my streaming session to survive brief network interruptions and
resume automatically without requiring me to restart either application, so that
a momentary Wi-Fi hiccup does not end my gaming session.

## Acceptance Criteria

1. If the connection drops, the client automatically attempts to reconnect for up to
   a configurable timeout (default: 30 seconds) before giving up
2. A successful reconnect resumes the stream without restarting RayHost
3. The user sees clear status feedback: "Connected", "Reconnecting…", "Disconnected"
4. Host and client negotiate stream parameters (resolution, frame rate, codec) once at
   connect time, so both sides agree on the stream format before media flows
5. Either side can cleanly end the session (e.g., `Ctrl+C` on the CLI) and the other
   side exits gracefully rather than hanging
6. Reconnect timeout is configurable via CLI or config file

## Technical Approach

Implement a session control channel alongside the media transport for keepalive,
capability exchange, and reconnect signalling. See [ADR-003](../adr/ADR-003.md) for the transport protocol
decision — the session control channel maps naturally to a reliable stream in the
chosen protocol. See [ADR-010](../adr/ADR-010.md) for the session resilience
protocol design and implementation plan.
