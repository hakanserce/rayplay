# ADR-008: Android Client UX Design and Platform Trade-offs

- **ADR ID:** ADR-008
- **Status:** Accepted
- **Blocks:** UC-022

## Context

The Android client (UC-022) must deliver the same gaming experience as the macOS client:
exclusive input mode with an external USB or Bluetooth keyboard and mouse connected to
the Android device. The user expects the game to respond exactly as if they were sitting
at the PC.

Android's input and security model differs significantly from macOS. Before committing
to UC-022's implementation, platform limitations and trade-offs must be understood and
documented so that the correct scope is agreed upon.

## Key Questions

1. Can Android achieve true exclusive input mode (mouse and keyboard fully captured,
   no system chrome, no cursor escaping to the status bar)?
2. Are external USB/BT keyboards and mice accessible with full keycode fidelity?
3. Can the mouse pointer be hidden and movement delivered as raw relative deltas?
4. What Android API levels and device constraints apply?

## Platform Analysis

### Mouse Input (external USB/BT mouse)

**USB (Option D path):** `UsbManager` + USB Host API reads raw HID reports directly
from the USB mouse, below the Android input stack. The OS never sees the events —
no pointer capture, no system chrome interference, no cursor to hide.

**Bluetooth (fallback):** `View.requestPointerCapture` (API 26+) hides the cursor and
delivers relative deltas via `MotionEvent.AXIS_RELATIVE_X / RELATIVE_Y`. Pointer
capture is released automatically when the app loses focus.

### Keyboard Input (external USB/BT keyboard)

**USB (Option D path):** `UsbManager` reads raw HID keyboard reports. The OS never
sees keystrokes — no `Esc`-as-back, no OEM modifier interception, full keycode fidelity.

**Bluetooth (fallback):** `KeyEvent`s delivered via Android input stack. Known gaps:
`Esc` may trigger back action (interceptable via `onKeyDown`); some OEM modifier
combinations may be intercepted.

### System Chrome Conflicts (Bluetooth / Option A only)

| Interaction | Mitigation | Suppressible? |
|-------------|-----------|---------------|
| Swipe from top (notification shade) | Immersive sticky mode delays it | No (transient) |
| Swipe from bottom/sides (gesture nav) | `BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE` | Transient only |
| Back gesture | `OnBackInvokedCallback` (API 33+) or `onBackPressed` | Yes |
| Home / recents button | None without device owner / kiosk mode | No |

These conflicts only apply when falling back to OS-level input capture (Bluetooth
peripherals). USB HID forwarding (Option D) is unaffected by all of the above.

## Decision

### Option A — Best-effort OS-level exclusive mode

Android pointer capture + immersive sticky mode. Near-exclusive input with known
limitations (home button, notification shade swipe).

**Accepted as Bluetooth fallback.** Used when USB peripherals are unavailable.

### Option B — Kiosk / Device Owner mode

Use Android Device Owner or MDM to suppress system chrome.

**Rejected.** Requires enterprise MDM setup — impractical for personal gaming.

### Option C — Rooted device

Full input suppression via root access.

**Rejected.** Out of scope for the target audience.

### Option D — USB HID Forwarding

Use Android USB Host API (`UsbManager`) to read raw HID reports from USB-connected
keyboard and mouse. Parse and forward to the host via the existing network protocol.
See ADR-006 Option D for the full cross-platform specification.

**Accepted as primary path for USB peripherals.** Provides true exclusive input on
Android without any of the OS-level restrictions — the Android input system never
sees the device events. No system chrome can interfere because there is nothing to
intercept.

## Input Strategy Summary

| Peripheral | Connection | Mechanism | Exclusive? |
|------------|------------|-----------|------------|
| Keyboard | USB | Option D (UsbManager raw HID) | Full |
| Mouse | USB | Option D (UsbManager raw HID) | Full |
| Keyboard | Bluetooth | Option A (Android InputEvent + intercept) | Best-effort |
| Mouse | Bluetooth | Option A (pointer capture) | Best-effort |

## Rendering

Per ADR-005, `winit` + `wgpu` via `android-activity` handles the Android surface.
MediaCodec decodes into a `SurfaceTexture` imported as a `wgpu` external texture on
the Vulkan backend.

## Implementation Requirements

- **Minimum API level:** Android 8 (API 26) for pointer capture (Bluetooth fallback).
  USB Host API is available from API 12 — no additional minimum version constraint.
- **USB permission:** `UsbManager.requestPermission()` must be called once per device.
  The app must handle the permission dialog and gracefully degrade to Bluetooth
  fallback if denied.
- **Immersive sticky mode (Bluetooth fallback):** `WindowInsetsController.hide(systemBars())`
  + `BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE` on API 30+; `SYSTEM_UI_FLAG_IMMERSIVE_STICKY`
  on API 26–29.
- **Back gesture interception:** `OnBackInvokedCallback` (API 33+) or `onBackPressed`
  override on older APIs to prevent accidental session exit.
- **No on-screen controls:** The UX assumes an external keyboard and mouse.
  On-screen gamepad overlays are explicitly out of scope for UC-022.

## Consequences

- **UC-022 ACs must reflect the two-tier model:** Full exclusive mode for USB
  peripherals; best-effort for Bluetooth. ACs must document the Bluetooth limitations
  (home button, notification shade) explicitly.

- **Shared HID parsing:** The HID usage-to-scancode translation table (ADR-006) is
  shared between the macOS and Android Option D paths. It lives in `rayplay-input`
  and has no platform-specific code.

- **Input translation:** Android NDK / USB HID keycodes are mapped to the same
  serialised `InputEvent` wire format as macOS — a single host-side injection path
  handles both clients.

- **Revisit trigger:** If a future Android API level introduces a gaming-focused input
  suppression API for Bluetooth devices (analogous to USB HID's OS bypass), revisit
  this ADR to upgrade the Bluetooth fallback.
