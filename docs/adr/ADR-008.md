# ADR-008: Android Client UX Design and Platform Trade-offs

- **ADR ID:** ADR-008
- **Status:** Proposed
- **Blocks:** UC-022

## Context

The Android client (UC-022) must deliver the same gaming experience as the macOS client:
exclusive input mode with an external USB or Bluetooth keyboard and mouse. This is the
target UX — the user physically connects peripherals to their Android device and expects
the game to respond exactly as if they were sitting at the PC.

However, Android's input and security model differs significantly from macOS. Before
committing to UC-022's implementation, these platform limitations and trade-offs must
be understood and documented so that the correct scope is agreed upon.

## Key Questions

1. Can Android achieve true exclusive input mode (mouse and keyboard fully captured,
   no system chrome, no cursor escaping to the status bar)?
2. Are external USB/BT keyboards and mice accessible with full keycode fidelity
   (including gaming keys, modifier combinations, media keys)?
3. Can the mouse pointer be hidden and mouse movement delivered as raw relative deltas?
4. What Android API levels and device constraints apply?

## Platform Analysis

### Mouse Input on Android (external USB/BT mouse)

**`View.onGenericMotionEvent` + `InputDevice.SOURCE_MOUSE`:**
Android delivers pointer events from an external mouse to the focused view. Raw relative
deltas are available via `MotionEvent.AXIS_RELATIVE_X / RELATIVE_Y` (API 24+, Android 7).
This is sufficient for gaming mouse input without absolute coordinate mapping.

**Pointer capture (`View.requestPointerCapture`, API 26+):**
Available on Android 8+. When a view has pointer capture, the cursor is hidden and all
mouse events are delivered as relative deltas to that view. This is the Android equivalent
of macOS cursor lock. Critical limitation: pointer capture is released automatically when
the app loses focus (e.g., notification shade, back gesture).

**Conclusion:** Pointer capture on Android 8+ provides functionally equivalent exclusive
mouse mode for a focused app. Back gesture / status bar swipes remain a risk — see
trade-offs below.

### Keyboard Input on Android (external USB/BT keyboard)

External keyboards on Android deliver `KeyEvent`s with standard Android keycodes, which
map well to HID keycodes. Most gaming keys work. Known gaps:
- `Esc` key may trigger Android's back action (configurable in `onKeyDown`)
- Some modifier combinations are intercepted by the system (e.g., Alt+Tab on some OEMs)
- No equivalent to macOS's ability to suppress all system shortcuts at the OS level

**Conclusion:** External keyboard input is largely equivalent to macOS, with a small set
of system-reserved key combinations that cannot be fully suppressed on unrooted devices.

### Exclusive Input Mode — System Chrome Conflicts

On Android, the following system interactions can break exclusive mode:
- **Swipe from top**: Notification shade (cannot be fully disabled on stock Android)
- **Swipe from bottom / sides**: Gesture navigation (can be mitigated by setting
  `WindowInsetsController.BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE` + immersive mode)
- **Back gesture**: Interceptable via `OnBackInvokedCallback` (API 33+) or
  `onBackPressed` override (older APIs)
- **Home button / recents**: Cannot be suppressed without device owner / kiosk mode

**Mitigation:** Immersive sticky mode (`SYSTEM_UI_FLAG_IMMERSIVE_STICKY` or
`WindowInsetsController` equivalent) hides status bar and navigation bar. Swipes show
them transiently but they auto-hide. This is the best achievable on a standard device.

## Decision

> **To be decided.** Proposed approach:
>
> **Option A — Best-effort exclusive mode (recommended)**
> Use Android pointer capture + immersive sticky mode. This gives near-exclusive input
> for gaming with the following known limitations:
> - Top-edge swipe can still open notification shade (unavoidable)
> - Home/recents cannot be fully suppressed
> - A small number of system key combinations may not be capturable
>
> Document these limitations clearly in the app. The experience is equivalent to macOS
> for the vast majority of gameplay scenarios.
>
> **Option B — Kiosk / Device Owner mode**
> Use Android Device Owner or dedicated device mode to fully suppress system chrome.
> Requires enterprise MDM setup — impractical for a personal gaming use case.
>
> **Option C — Rooted device**
> Root access allows full input suppression. Out of scope for the target audience.
>
> **Recommended:** Option A. Ship with best-effort exclusive mode, document the
> limitations (top-swipe, home button), and let users decide if it meets their needs.

## Consequences

- UC-022 ACs must reflect Option A's limitations explicitly — "exclusive input mode
  on Android is best-effort; system gestures from screen edges may not be suppressible."
- Minimum target API level: Android 8 (API 26) for pointer capture.
- Immersive sticky mode implementation is required in the Android rendering layer.
- `Esc` and system-reserved key combinations must be documented as known gaps.
- No on-screen overlay controls are required — the UX assumes an external keyboard and
  mouse are connected. On-screen controls are explicitly out of scope.
- This ADR should be revisited if Android adds a gaming-focused input API in a future
  API level.
