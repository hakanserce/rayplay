# ADR-009: Platform-Neutral Fallback Pipeline Strategy

- **ADR ID:** ADR-009
- **Status:** Accepted
- **Blocks:** [UC-026](../uc/UC-026.md)
- **Amends:** [ADR-000](ADR-000.md) (Principle #2: hardware-only stance)
- **Supersedes:** [ADR-004](ADR-004.md) rejection of FFmpeg (for fallback scope only)

## Context

RayPlay's current pipeline is optimized for specific platforms: NVENC hardware
encoding on Windows hosts and IOSurface/VideoToolbox hardware decoding on macOS
clients. This makes development and testing impossible on other platforms (e.g.,
Linux, macOS as host, Windows as client) and limits RayPlay's reach.

A platform-neutral fallback is needed for screen capture, video encoding, video
decoding, and frame rendering ‚ÄĒ one that works on any desktop OS at the cost of
higher latency. This ADR decides the concrete libraries, APIs, and architecture
for each stage of the fallback pipeline.

ADR-000 Principle #2 states "hardware acceleration everywhere; software fallbacks
are not a goal." This ADR amends that stance: a software fallback is now a P1
requirement (UC-026) for developer productivity and broader platform reach, while
the optimized hardware path remains the primary target for production use.

## Key Questions

1. What software encoder should be used (FFmpeg/libx264, pure Rust codec, other)?
2. What cross-platform screen capture mechanism should be used?
3. What software decoder should be used on the client side?
4. What rendering approach works across all desktop platforms?
5. How should the optimized and fallback paths coexist ‚ÄĒ compile-time features,
   runtime detection, or both?
6. What is the acceptable latency and quality trade-off for the fallback path?

## Decision

### 1. Screen Capture: `scrap` crate

Use the [`scrap`](https://crates.io/crates/scrap) crate (MIT license) for
cross-platform screen capture.

- Uses DXGI Desktop Duplication on Windows, CGDisplay on macOS, X11 SHM /
  PipeWire on Linux
- Battle-tested in RustDesk (major remote desktop project)
- Returns raw BGRA pixel data ‚ÄĒ maps directly to `CapturedFrame`
- Estimated latency: 5‚Äď10ms per frame

**Rejected alternatives:**
- `xcap`: Newer, less battle-tested. Revisit if `scrap`'s Wayland support
  proves insufficient.
- FFmpeg `avdevice`: Overkill for capture alone; pulls in libavformat/libavdevice.

### 2. Software Encoding: `openh264` (default) + `ffmpeg-next` (opt-in)

**Default:** Use the [`openh264`](https://crates.io/crates/openh264) crate
(BSD-2-Clause) which wraps Cisco's OpenH264 library.

- Auto-downloads Cisco's pre-built binary at build time ‚ÄĒ zero system dependency
- Supports H.264 Baseline/Constrained Baseline profile
- Estimated encoding latency: 15‚Äď25ms at 1080p30 (fastest preset)
- BSD-2-Clause license ‚ÄĒ no copyleft concern

**Opt-in:** Behind a `ffmpeg-fallback` Cargo feature flag, enable `ffmpeg-next`
with `libx264` for users who accept the GPL-2.0 dependency.

- `libx264 -preset ultrafast -tune zerolatency`: 8‚Äď12ms at 1080p30
- Enables 60fps fallback on capable hardware
- ADR-004 rejected FFmpeg for the optimized path (buffering, LGPL). Those
  concerns do not apply here: `-tune zerolatency` eliminates B-frame buffering,
  and the fallback is a different quality tier with relaxed latency targets.

The `Codec` enum gains an `H264` variant alongside the existing `Hevc`.

### 3. Software Decoding: `openh264` (default) + `ffmpeg-next` (opt-in)

**Default:** Use `openh264` for decoding. Pairs with the encoder, BSD-licensed,
and H.264 software decode at 1080p30 is computationally trivial (2‚Äď5ms).

**Opt-in:** When `ffmpeg-fallback` feature is enabled, use `ffmpeg-next` for
decoding as well. This avoids codec mismatch and supports HEVC decode.

### 4. Rendering: `wgpu` (existing ‚ÄĒ no changes needed)

The existing `WgpuRenderer` already handles BGRA8 CPU frames via
`wgpu::Queue::write_texture` (see ADR-005). This path works on all desktop
platforms (Vulkan, Metal, DX12). No additional rendering work is required.

**Future consideration:** Add `softbuffer` behind a feature flag for headless /
no-GPU environments (CI, VMs). This is low priority since most desktops have a GPU.

### 5. Coexistence Strategy: Runtime Detection + Compile-Time Feature Gates

**Feature flags:**
- `fallback` (default-enabled): Compiles the `scrap` + `openh264` fallback path
- `ffmpeg-fallback` (opt-in): Adds `ffmpeg-next`/`libx264` as an alternative
  software backend

**Runtime behavior:**
- Factory functions (`create_encoder`, `create_capturer`, `create_decoder`)
  try the optimized hardware path first
- If the hardware path returns `UnsupportedPlatform`, automatically fall back
  to the software implementation
- CLI flag `--software` forces the fallback path regardless of platform
  capability (UC-026 AC #4)

This leverages the existing trait-object architecture (`VideoEncoder`,
`VideoDecoder`, `ScreenCapturer`, `Renderer`) with no structural changes.

### 6. Latency Budget

| Stage              | Optimized Path | Fallback (openh264) | Fallback (libx264) |
|--------------------|----------------|---------------------|--------------------|
| Screen capture     | <1ms (DXGI)    | 5‚Äď10ms (scrap)      | 5‚Äď10ms (scrap)     |
| Encode             | <5ms (NVENC)   | 15‚Äď25ms             | 8‚Äď12ms             |
| Network            | <1ms           | <1ms                | <1ms               |
| Decode             | <3ms (VTB)     | 2‚Äď5ms               | 2‚Äď5ms              |
| Render             | <2ms (IOSurf)  | 2‚Äď3ms (wgpu CPU)    | 2‚Äď3ms (wgpu CPU)   |
| **Total**          | **<12ms**      | **25‚Äď44ms**         | **18‚Äď31ms**        |
| **Target FPS**     | 60fps          | 30fps               | 30‚Äď60fps           |

The default `openh264` fallback is borderline for 30fps (33ms budget) at 1080p
but achievable on modern quad-core CPUs. The `libx264` opt-in path comfortably
hits 30fps and can reach 60fps on capable hardware.

## Migration Path

Implementation is split into five phases. Each phase is a separate UC / PR that
can be reviewed and merged independently.

### Phase 1: Shared Plumbing (UC-026a)

Prepare the crate for multiple backends without changing any runtime behavior.

**1a. Add `H264` codec variant**

File: `crates/rayplay-video/src/encoder.rs`

```rust
pub enum Codec {
    Hevc,
    H264,  // ‚Üź new
}
```

Update `compute_auto_bitrate` to account for H.264's lower compression
efficiency (~15x vs HEVC's ~20x). Update all `match` arms that handle `Codec`.

**1b. Add `create_decoder` factory function**

File: `crates/rayplay-video/src/decoder.rs`

Currently there is no `create_decoder` factory ‚ÄĒ `VtDecoder` is constructed
directly. Add a factory mirroring `create_encoder` / `create_capturer`:

```rust
pub fn create_decoder(codec: Codec) -> Result<Box<dyn VideoDecoder>, VideoError> {
    #[cfg(target_os = "macos")]
    { VtDecoder::new(codec).map(|d| Box::new(d) as _) }

    #[cfg(not(target_os = "macos"))]
    { Err(VideoError::UnsupportedPlatform) }
}
```

Export it from `crates/rayplay-video/src/lib.rs`.

**1c. Add `fallback` and `ffmpeg-fallback` feature flags**

File: `crates/rayplay-video/Cargo.toml`

```toml
[features]
fallback        = ["dep:scrap", "dep:openh264"]
ffmpeg-fallback = ["dep:ffmpeg-next"]
default         = ["fallback"]

[dependencies]
scrap     = { version = "0.5", optional = true }
openh264  = { version = "0.6", optional = true }

[dependencies.ffmpeg-next]
version  = "7"
optional = true
```

No implementations yet ‚ÄĒ just dependency wiring.

### Phase 2: Software Screen Capture (UC-026b)

**2a. Implement `ScrapCapturer`**

New file: `crates/rayplay-video/src/scrap_capture.rs`

```rust
pub struct ScrapCapturer { /* scrap::Capturer, resolution, config */ }
impl ScreenCapturer for ScrapCapturer { ... }
```

Wraps `scrap::Capturer`. Converts `scrap::Frame` (BGRA) ‚Üí `CapturedFrame`.
Guard with `#[cfg(feature = "fallback")]`.

**2b. Wire into `create_capturer` factory**

File: `crates/rayplay-video/src/capture.rs`

Change the non-Windows branch from returning `UnsupportedPlatform` to attempting
`ScrapCapturer` first (when `fallback` feature is enabled):

```rust
#[cfg(not(target_os = "windows"))]
{
    #[cfg(feature = "fallback")]
    { ScrapCapturer::new(config).map(|c| Box::new(c) as _) }

    #[cfg(not(feature = "fallback"))]
    { Err(CaptureError::UnsupportedPlatform) }
}
```

On Windows, `DxgiCapture` remains the primary path. Add a `force_software`
parameter (or a separate `create_software_capturer` function) for the
`--software` CLI flag.

**2c. Register module**

File: `crates/rayplay-video/src/lib.rs` ‚ÄĒ add `pub mod scrap_capture;` behind
the feature gate.

### Phase 3: Software Codec ‚ÄĒ OpenH264 (UC-026c)

**3a. Implement `OpenH264Encoder`**

New file: `crates/rayplay-video/src/openh264_enc.rs`

```rust
pub struct OpenH264Encoder { /* openh264::encoder::Encoder, config */ }
impl VideoEncoder for OpenH264Encoder { ... }
```

- Accepts `EncoderInput::Cpu` only (return `VideoError::EncodingFailed` for
  `GpuTexture`)
- Uses `openh264::encoder::Encoder` with `EncoderConfig` mapped to OpenH264
  parameters (Baseline profile, fastest complexity, `CAMERA_VIDEO_REAL_TIME`
  usage type)
- Outputs NAL units wrapped in `EncodedPacket`

**3b. Implement `OpenH264Decoder`**

New file: `crates/rayplay-video/src/openh264_dec.rs`

```rust
pub struct OpenH264Decoder { /* openh264::decoder::Decoder */ }
impl VideoDecoder for OpenH264Decoder { ... }
```

- Decodes H.264 NAL units ‚Üí YUV420 ‚Üí convert to BGRA8 `DecodedFrame`
  (via `DecodedFrame::new_cpu`)
- YUV‚ÜíBGRA conversion: use a simple inline converter or the `color_conv` utility
  from `openh264` (avoid pulling in a heavy imaging crate for one conversion)

**3c. Wire into factory functions**

File: `crates/rayplay-video/src/encoder.rs` ‚ÄĒ in `create_encoder`:

```rust
#[cfg(not(target_os = "windows"))]
{
    #[cfg(feature = "fallback")]
    {
        use crate::openh264_enc::OpenH264Encoder;
        let config = config.with_codec(Codec::H264); // force H264 for software
        OpenH264Encoder::new(config).map(|e| Box::new(e) as _)
    }
    #[cfg(not(feature = "fallback"))]
    { Err(VideoError::UnsupportedPlatform) }
}
```

File: `crates/rayplay-video/src/decoder.rs` ‚ÄĒ in `create_decoder`, add
the same pattern for the non-macOS branch.

**3d. Register modules**

File: `crates/rayplay-video/src/lib.rs` ‚ÄĒ add:
```rust
#[cfg(feature = "fallback")]
pub mod openh264_enc;
#[cfg(feature = "fallback")]
pub mod openh264_dec;
```

### Phase 4: CLI Integration (UC-026d)

**4a. Add `--software` flag**

File: `crates/rayplay-cli/src/main.rs` (or wherever clap args are defined)

Add a `--software` boolean flag. When set, the CLI passes a `force_software:
true` option to the factory functions, bypassing hardware detection.

**4b. Implement `force_software` in factory functions**

Add a `PipelineMode` enum to `crates/rayplay-video/src/capture.rs` and
`encoder.rs`:

```rust
pub enum PipelineMode {
    Auto,     // try hardware first, fall back to software
    Software, // force software path
}
```

Update `create_capturer`, `create_encoder`, `create_decoder` signatures to
accept `PipelineMode`. When `Software`, skip the hardware attempt entirely.

### Phase 5: FFmpeg Opt-In Backend (UC-026e ‚ÄĒ optional, lower priority)

**5a. Implement `FfmpegEncoder`**

New file: `crates/rayplay-video/src/ffmpeg_enc.rs`
Guard: `#[cfg(feature = "ffmpeg-fallback")]`

- Uses `ffmpeg_next::codec::encoder::video::Video`
- Configures `libx264` with `-preset ultrafast -tune zerolatency`
- Supports both `H264` and `Hevc` (via `libx265`) codecs

**5b. Implement `FfmpegDecoder`**

New file: `crates/rayplay-video/src/ffmpeg_dec.rs`
Guard: `#[cfg(feature = "ffmpeg-fallback")]`

**5c. Update factory preference order**

When `ffmpeg-fallback` is enabled, the factory functions prefer FFmpeg over
OpenH264 for the software path (better performance). When both features are
enabled, the precedence is: hardware ‚Üí FFmpeg software ‚Üí OpenH264 software.

### Summary: Files Changed Per Phase

| Phase | New Files | Modified Files |
|-------|-----------|---------------|
| 1 | ‚ÄĒ | `encoder.rs`, `decoder.rs`, `lib.rs`, `Cargo.toml` |
| 2 | `scrap_capture.rs` | `capture.rs`, `lib.rs` |
| 3 | `openh264_enc.rs`, `openh264_dec.rs` | `encoder.rs`, `decoder.rs`, `lib.rs` |
| 4 | ‚ÄĒ | CLI crate, `capture.rs`, `encoder.rs`, `decoder.rs` |
| 5 | `ffmpeg_enc.rs`, `ffmpeg_dec.rs` | `encoder.rs`, `decoder.rs`, `lib.rs`, `Cargo.toml` |

## Consequences

### Positive

- **Developer productivity:** All contributors can build, test, and debug RayPlay
  on any desktop OS without specialized hardware
- **Broader platform reach:** Linux and cross-platform streaming become possible
- **CI/CD:** Integration tests can run against the software pipeline in GitHub
  Actions without GPU runners
- **License-clean default:** The `openh264` default path is BSD-licensed with no
  copyleft obligations
- **Minimal architectural impact:** Existing trait boundaries and factory functions
  accommodate the fallback with no structural changes

### Negative

- **New dependencies:** `scrap` and `openh264` add to the dependency tree
- **Latency regression:** The fallback path is 2‚Äď4x slower than the optimized
  path. Users must understand this is not the target experience.
- **OpenH264 quality:** Baseline profile has lower compression efficiency than
  HEVC. Visual quality at equivalent bitrates will be noticeably worse.
- **Maintenance surface:** Two codec backends (three with FFmpeg opt-in) means
  more code paths to test and maintain
- **GPL opt-in risk:** Users enabling `ffmpeg-fallback` must understand the
  GPL-2.0 license implications for distribution
