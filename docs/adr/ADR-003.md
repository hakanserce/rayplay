# ADR-003: Streaming Protocol Selection

- **ADR ID:** ADR-003
- **Status:** Proposed
- **Blocks:** UC-003, UC-009, UC-010, UC-012, UC-015

## Context

RayPlay requires a transport protocol for video frames, audio packets, and input events.
The protocol choice is the single most impactful architectural decision for latency. Key
requirements:

- **Video:** Large frames (10–100 KB), loss-tolerant (a dropped frame is better than
  stalled delivery), order-sensitive within a GOP, time-sensitive
- **Input:** Tiny packets (<64 bytes), ultra-low latency, some loss-tolerance
  (stale mouse position is OK; missed key-down is not)
- **Audio:** Small frames (~500 bytes), jitter-sensitive, some loss-tolerance
- **Control/Session:** Reliable, low-frequency signalling (handshake, keepalive)
- **Security:** Encryption required (see ADR-007)

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — Raw UDP (custom framing)**
> Minimal overhead. Full control over retransmission, FEC, and congestion. Requires
> implementing reliability where needed (for control messages). Proven by Sunshine/Moonlight.
>
> **Option B — QUIC (via `quinn` crate)**
> UDP-based, multiplexed streams, built-in TLS 1.3, congestion control. Supports
> unreliable datagrams (RFC 9221) for video/input. Control messages use reliable streams.
> Good fit: single connection carries all channels with appropriate reliability per stream.
>
> **Option C — WebRTC (via `webrtc-rs`)**
> Standard protocol for browser-based clients. Higher overhead, complex signalling (ICE,
> SDP). Justified only if browser support is a goal — it is not for this project.
>
> **Option D — Custom UDP + DTLS**
> Raw UDP with DTLS for encryption. Decouples encryption from transport logic. More
> control than QUIC but more implementation work.
>
> **Recommended starting point:** Option B (QUIC via `quinn`) — eliminates the need to
> implement reliability, congestion, and encryption from scratch, while supporting
> unreliable datagrams for latency-sensitive channels. Fallback: Option A if QUIC
> overhead proves measurably worse than raw UDP on LAN benchmarks.

## Consequences

- QUIC's unreliable datagrams (RFC 9221) require `quinn` v0.11+ and must be benchmarked
  to verify that QUIC overhead stays under 1ms on LAN (see UC-003 AC #2).
- A custom framing layer is needed regardless of protocol to handle fragmentation of
  large video frames.
- The session control channel (UC-015) maps naturally to a QUIC reliable bidirectional
  stream, simplifying UC-015 implementation.
- ADR-007 (Security) must be resolved in concert with this ADR — QUIC subsumes TLS;
  raw UDP requires a separate encryption layer.
