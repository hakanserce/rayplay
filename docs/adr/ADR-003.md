# ADR-003: Streaming Protocol Selection

- **ADR ID:** ADR-003
- **Status:** Accepted
- **Blocks:** UC-003, UC-009, UC-010, UC-012, UC-015
- **Cascades to:** ADR-007 (accept Option A: QUIC + TLS 1.3 + SPAKE2)

## Context

RayPlay requires a transport protocol for video frames, audio packets, and input events.
The protocol choice is the single most impactful architectural decision for latency. Key
requirements per traffic type:

| Channel | Packet size | Loss tolerance | Ordering |
|---------|-------------|----------------|---------- |
| Video frames | 10â€“100 KB | High â€” drop frame, do not stall | Per-frame only |
| Audio frames | ~500 bytes | Moderate â€” brief glitch acceptable | Ordered |
| Mouse position | <32 bytes | High â€” stale position is fine | Latest wins |
| Keyboard / mouse buttons | <64 bytes | None â€” missed key-down is unacceptable | Strictly ordered |
| Control / session | <1 KB | None â€” reliability required | Ordered |

Additional constraints:
- Encryption required (see ADR-007)
- LAN-first: 100 Mbps+ throughput, <1 ms transport overhead per packet
- No internet infrastructure assumed (no STUN/TURN for P0 scope)

## Decision

### Option A â€” Raw UDP (custom framing)

Full control but requires implementing reliability, congestion control, and encryption
from scratch for the control and keyboard channels.

**Rejected.** The implementation cost is high and the result would be a bespoke,
under-tested protocol. Sunshine/Moonlight's approach was valid when QUIC was immature;
QUIC via `quinn` is now production-ready.

### Option C â€” WebRTC (`webrtc-rs`)

Browser interop protocol with ICE/SDP signalling overhead.

**Rejected.** No browser client is planned. Complexity is unjustified.

### Option D â€” Custom UDP + DTLS

UDP with DTLS for encryption, custom reliability on top.

**Rejected.** More implementation work than QUIC for equivalent security guarantees.
QUIC subsumes DTLS without the additional layering cost.

### Option B â€” QUIC via `quinn` â‰¥0.11

**Accepted.** QUIC is UDP-based with multiplexed streams, built-in TLS 1.3, and
unreliable datagram support (RFC 9221 via `quinn` â‰¥0.11). Each traffic type maps to
the QUIC primitive that matches its reliability requirement â€” no bespoke protocol needed.

## Channel Mapping

Each channel is bound to a specific QUIC primitive at connection setup:

| Channel | QUIC primitive | Rationale |
|---------|----------------|-----------|
| Video fragments | Unreliable datagrams (RFC 9221) | Loss-tolerant; avoids HOL blocking |
| Audio frames | Unreliable datagrams (RFC 9221) | Loss-tolerant; fits in one datagram |
| Mouse position | Unreliable datagrams (RFC 9221) | Latest value always supersedes |
| Keyboard + mouse buttons | Reliable unidirectional stream | Ordered, no loss tolerated |
| Control / session | Reliable bidirectional stream | Handshake, keepalive, reconnect |

One QUIC connection carries all channels. Streams are opened once per session and kept
alive; datagrams are sent per-event.

## Video Fragmentation

QUIC's max datagram size is bounded by PMTU minus QUIC/UDP headers â€” approximately
1 200 bytes on a standard LAN. Video frames (10â€“100 KB) must be manually fragmented
before being sent as unreliable datagrams and reassembled on the receiver.

Each video datagram carries a fixed 12-byte header:

```
 0               1               2               3
 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|                         frame_id (u32)                         |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|      frag_index (u16)         |       frag_total (u16)        |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|    channel (u8)   |  flags (u8)|        reserved (u16)        |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

Reassembly rules:
- Receiver allocates a fragment buffer per `frame_id`.
- If any fragment for a `frame_id` is missing when the *next* `frame_id` arrives,
  the incomplete frame is dropped and the decoder is notified (it will request an
  I-frame if needed).
- Fragment buffers are bounded to 4 in-flight frames to cap memory usage.

Audio datagrams use the same header with `frag_total = 1` (never fragmented).

## Keyboard / Mouse Button Stream

Keyboard key-down/key-up and mouse button events are sent on a QUIC reliable
unidirectional stream (client â†’ host). Each event is length-prefixed (2-byte LE length
+ payload). The stream is never closed during a session; events are written and flushed
immediately. The host processes events in arrival order.

Mouse *position* and scroll deltas use unreliable datagrams â€” they are independent
of the key stream and supersede each other naturally.

## Congestion Control

`quinn` defaults to New Reno congestion control. On a LAN with ample bandwidth, the
congestion window fills quickly and does not impose measurable latency. However,
unreliable datagrams bypass the congestion window entirely (RFC 9221 Â§5), which is
exactly the behaviour we want for video, audio, and mouse position.

The reliable key/control streams are the only channels subject to CC. Since their
throughput is tiny (<1 KB/s), CC will not be a bottleneck. If benchmarks reveal
unexpected latency on these streams, `quinn`'s `TransportConfig::initial_rtt` and
`max_idle_timeout` can be tuned without changing the protocol choice.

## Security

QUIC embeds TLS 1.3. This ADR's acceptance of Option B directly resolves ADR-007 in
favour of **ADR-007 Option A** (QUIC + TLS 1.3 + SPAKE2 for PIN pairing). A separate
encryption layer is not needed.

## Consequences

- **`quinn` â‰¥0.11** is required for RFC 9221 datagram support. Pin this in `Cargo.toml`.
  The `quinn` `Endpoint` is owned by `rayplay-network`; other crates interact via
  channel abstractions, never touching `quinn` types directly.

- **Framing layer in `rayplay-network`:** The `VideoFragmenter` (sender) and
  `VideoReassembler` (receiver) structs handle fragmentation/reassembly. The
  `InputChannel` struct multiplexes keyboard events onto the reliable stream and mouse
  events onto datagrams. All are internal to `rayplay-network`.

- **`NetworkTransport` trait in `rayplay-core`:** A thin trait exposes `send_video`,
  `send_audio`, `send_input`, `send_control`, and matching receive methods. This keeps
  `rayplay-video` and `rayplay-input` crates independent of `quinn`.

- **Benchmarking requirement:** Before UC-003 is done, a Criterion benchmark must verify
  that the QUIC datagram path adds <1 ms overhead per video packet at 1080p60 on a LAN.
  Measure with `quinn`'s unreliable datagram API, not streams.

- **CI / cross-platform:** `quinn` is pure Rust and cross-compiles to Windows and macOS.
  No platform-specific build steps required for the network crate.

- **Future: NAT traversal** â€” out of scope for P0 (LAN-only). If a relay or direct WAN
  connection is needed later, `quinn`'s `Endpoint` supports connection migration and
  can be extended with STUN without changing the channel abstraction.

- **Future: QUIC stream per video frame** â€” an alternative to datagrams where each frame
  opens a new unidirectional stream and is reset if superseded. This avoids manual
  fragmentation but has higher per-frame overhead. Defer evaluation to post-P0
  benchmarking.
