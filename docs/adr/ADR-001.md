# ADR-001: Zero-Copy Graphics Capture/Encode/Send Architecture

- **ADR ID:** ADR-001
- **Status:** Proposed
- **Blocks:** UC-001, UC-002, UC-003

## Context

To achieve sub-16ms glass-to-glass latency, frames must flow from the GPU capture stage to
the network send stage with minimal memory copies. Each copy adds latency and consumes
memory bandwidth. The key question is: how do we keep frame data on the GPU (or in
pinned/shared memory) from capture through NVENC encoding to the network send buffer?

Key constraints:
- Capture via DXGI Desktop Duplication API (Windows, DirectX)
- Encoding via NVENC (CUDA-based, operates on GPU memory)
- Network send requires CPU-accessible buffers or kernel bypass (RDMA)
- Must support 60 fps at 1080p/4K without saturating the PCIe bus

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — CUDA-Pinned Host Memory**
> DXGI captures to a DirectX texture. CopyResource to a CUDA-registered pinned buffer.
> NVENC reads directly from pinned memory. Bitstream is DMA'd out by NVENC into a
> CPU-visible ring buffer for network send. One copy: GPU texture → pinned host memory.
>
> **Option B — NVENC Direct from DXGI Texture**
> Register the DXGI shared surface with NVENC via `NvEncRegisterResource`. NVENC reads
> directly from the DirectX texture without any copy. Bitstream lands in a host-mapped
> output buffer for send. Zero copies on the input path.
>
> **Option C — CUDA Interop + GPU-Direct**
> Full CUDA interop pipeline; NVENC output written to GPU memory, GPU-Direct RDMA to NIC.
> Eliminates all PCIe transfers. Requires RDMA-capable NIC — impractical for consumer hardware.
>
> **Recommended starting point:** Option B (NVENC Direct from DXGI Texture) minimises
> copies and is well-supported on consumer Nvidia hardware. Option A is the fallback if
> DXGI/NVENC interop proves unreliable across driver versions.

## Consequences

- **Option A:** Predictable, driver-independent, but adds one PCIe copy per frame.
  At 60 fps/4K this is ~3–4 GB/s of PCIe bandwidth — feasible but significant.
- **Option B:** Zero input-path copies. NVENC + DXGI interop is well-documented but
  requires careful resource lifetime management across D3D/NVENC contexts.
- **Option C:** Optimal throughput but impractical for the target hardware; defer to future ADR.

The chosen approach must be validated by a benchmark (Criterion) measuring encoding
latency at 1080p60 and 4K60 before UC-002 is considered done.
