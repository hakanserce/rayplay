# ADR-001: Zero-Copy Graphics Capture/Encode/Send Architecture

- **ADR ID:** ADR-001
- **Status:** Accepted
- **Blocks:** [UC-001](../uc/UC-001.md), [UC-002](../uc/UC-002.md), [UC-003](../uc/UC-003.md)

## Context

To achieve sub-16ms glass-to-glass latency, frames must flow from the GPU capture stage to
the network send stage with minimal memory copies. Each copy adds latency and consumes
memory bandwidth. The key question is: how do we keep frame data on the GPU (or in
pinned/shared memory) from capture through NVENC encoding to the network send buffer?

Key constraints:
- Capture via DXGI Desktop Duplication API (Windows, DirectX)
- Encoding via NVENC (CUDA-based, operates on GPU memory)
- Network send requires CPU-accessible buffers (LAN, no RDMA hardware)
- Must support 60 fps at 1080p/4K without saturating the PCIe bus

## Decision

### Option C — CUDA Interop + GPU-Direct (RDMA)

Full CUDA pipeline with GPU-Direct RDMA to the NIC. Eliminates all PCIe transfers.

**Rejected.** Requires RDMA-capable NIC. Impractical for consumer hardware (RTX 2060
target). Deferred to a future ADR if dedicated streaming hardware becomes relevant.

### Option A — CUDA-Pinned Host Memory

DXGI captures to a DirectX texture. `CopyResource` to a CUDA-registered pinned buffer.
NVENC reads from pinned memory. Bitstream is DMA'd into a CPU-visible ring buffer.
One PCIe copy: GPU texture → pinned host memory (~3–4 GB/s at 4K60).

**Accepted as fallback.** Predictable, driver-independent. Activate if Option B proves
unreliable across driver versions in practice.

### Option B — NVENC Direct from DXGI Texture

Register the DXGI shared surface with NVENC via `NvEncRegisterResource`. NVENC reads
directly from the DirectX texture — zero copies on the input path. The output bitstream
lands in a pre-allocated host-mapped ring buffer for the network layer to read.

**Accepted as primary.** Zero input-path copies. Well-supported on consumer Nvidia
hardware (RTX 2060+, NVENC API ≥9.0, driver ≥418.81).

---

## DXGI → NVENC Registration Lifecycle

The zero-copy path requires careful resource lifetime management:

1. **Startup:** Create a single `ID3D11Device` shared by both `DxgiCapture` and
   `NvencEncoder`. Open an NVENC encode session (`NvEncOpenEncodeSessionEx`) passing
   the shared D3D11 device handle.
2. **Texture registration:** For each `ID3D11Texture2D` acquired from Desktop Duplication,
   call `NvEncRegisterResource` with `NV_ENC_INPUT_RESOURCE_TYPE_DIRECTX`. Cache the
   returned `NV_ENC_REGISTERED_PTR` for the lifetime of the texture.
3. **Per-frame encode:**
   a. `NvEncMapInputResource` — maps the registered texture into an NVENC input buffer.
   b. `NvEncEncodePicture` — encodes from the mapped buffer (GPU-to-GPU).
   c. `NvEncUnmapInputResource` — releases the map.
   d. Only after unmap: `ReleaseFrame` returns the DXGI texture to Desktop Duplication.
4. **Teardown:** `NvEncUnregisterResource` for all registered textures before releasing
   the D3D11 device.

The DXGI texture must not be released to Desktop Duplication while NVENC holds a map.
The capture loop must enforce this ordering.

## Output Buffer

NVENC writes the encoded bitstream into a pre-allocated `NV_ENC_LOCK_BITSTREAM` output
buffer (host-mapped, pinned memory) created via `NvEncCreateBitstreamBuffer`. This buffer
is owned by the NVENC session and reused across frames. A ring of `2 × max_in_flight_frames`
output buffers prevents stalls while the network layer drains them.

## Thread Model

D3D11 and NVENC have thread-affinity requirements. The entire capture → register → encode
→ output path runs on a single dedicated OS thread (not a tokio task). The thread
communicates with the async network layer via a bounded `tokio::sync::mpsc` channel
carrying bitstream buffer references.

## Fallback Trigger

Switch from Option B to Option A if driver compatibility issues with
`NvEncRegisterResource` are observed on target hardware during integration testing, or
if benchmarks show no measurable latency improvement over Option A. The fallback must be
transparent — the encode session interface must not expose which path is active.

## Consequences

- **Shared device:** The `ID3D11Device` must be created at startup and passed to both
  `DxgiCapture` and `NvencEncoder`. Neither subsystem creates its own device.

- **Implementation gap (tracked in issue #31):** The current `dxgi_capture.rs` uses
  `CopyResource` + CPU `map_and_read` (Option A, worse). The `NvencEncoder` is a stub.
  Both must be replaced before the <16ms latency target is achievable.

- **Benchmark requirement:** Before [UC-002](../uc/UC-002.md) is done, a Criterion benchmark must confirm
  <5 ms encoding latency at 1080p60 and 4K60 with Option B active.

- **Driver check:** `build.rs` must emit a runtime warning if the detected NVENC API
  version is <9.0, and automatically activate the Option A fallback path.
