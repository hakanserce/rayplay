# ADR-007: Security Model (Encryption, Authentication, Trust)

- **ADR ID:** ADR-007
- **Status:** Accepted
- **Blocks:** [UC-016](../uc/UC-016.md)
- **Resolved by:** [ADR-003](ADR-003.md) (QUIC accepted → Option A follows directly)

## Context

RayPlay streams live video, audio, and input across the network. On a shared or public
network, an unauthenticated stream is a significant security and privacy risk. The
security model must address:

1. **Authentication:** How does the host verify that a client is authorised?
2. **Encryption:** How is stream data protected from eavesdropping?
3. **Trust persistence:** How are previously-authorised clients remembered?
4. **Revocation:** How can a user remove a previously-trusted client?

Constraints:
- No PKI infrastructure (no certificate authority, no DNS)
- PIN-based pairing flow required ([UC-016](../uc/UC-016.md))
- Must work on LAN without internet connectivity
- Cryptographic implementation must use audited, maintained Rust crates
- [ADR-003](ADR-003.md) accepted QUIC (`quinn`) as the transport — TLS 1.3 is built in

## Decision

### Option A — QUIC + TLS 1.3 + SPAKE2 pairing

QUIC (`quinn`) embeds `rustls` for TLS 1.3. For PIN-based pairing, use SPAKE2 (via the
`spake2` crate) during the initial handshake. After pairing, the client's public key is
stored on the host as a trusted identity. Subsequent connections authenticate via a
challenge-response protocol over the control channel using the stored ed25519 key pair.

**Accepted.** [ADR-003](ADR-003.md) already chose QUIC, so TLS 1.3 is free. SPAKE2 is a
well-studied, audited PAKE protocol. Reusing the transport's security layer avoids a
separate encryption stack entirely.

### Option B — Noise Protocol Framework (`snow` crate)

Noise `XX` handshake for mutual authentication and encryption, layered over any
transport.

**Rejected.** QUIC already provides TLS 1.3 with equivalent security guarantees.
Adding Noise on top would be redundant encryption. Noise remains a valid choice only
if the transport were raw UDP (which ADR-003 rejected).

### Option C — Pre-shared Key (PSK) + AES-GCM

Derive a shared secret from the PIN via HKDF; use it for AES-256-GCM.

**Rejected.** PSK offers no forward secrecy and requires re-pairing if the key is
compromised. TLS 1.3 + SPAKE2 provides forward secrecy and a better security model
with comparable simplicity.

## Pairing Flow

1. **First connection:** The host generates a 6-digit PIN and displays it on screen.
2. The client prompts the user to enter the PIN.
3. The client sends a `ClientHello` message declaring its intent (pairing or authentication).
4. For pairing, both sides execute a SPAKE2 exchange over the QUIC control stream:
   - Each side commits to the PIN-derived password (`spake2::Spake2::start_*`)
   - Exchange messages; derive a shared `session_key`
   - Verify the key matches on both sides via a MAC confirmation message
5. On success, the client's ed25519 public key is stored in the host's trust database.
6. **Subsequent connections:** The client sends a `ClientHello` declaring auth intent. The host sends a random nonce challenge. The client signs the nonce with its ed25519 key and returns the signature along with its public key. The host verifies the signature and checks the public key against its trust database. No PIN needed.

Pairing must complete in under 10 seconds ([UC-016](../uc/UC-016.md) AC #6). On a LAN, SPAKE2 adds
one round-trip — well within budget.

## Trust Database

Paired client identities are stored on the host as a JSON file in the platform config
directory (`%APPDATA%\RayPlay\trusted_clients.json` on Windows). Each entry contains:

```json
{
  "client_id": "<human-readable label>",
  "public_key": "<base64-encoded ed25519 public key>",
  "paired_at": "<ISO 8601 timestamp>",
  "last_seen": "<ISO 8601 timestamp>"
}
```

The file is created with restricted permissions (owner read/write only). Private keys
are never stored on the host — only client public keys.

## Cryptographic Crates

| Purpose | Crate | Rationale |
|---------|-------|-----------|
| TLS 1.3 | `rustls` (via `quinn`) | Audited, widely deployed, no OpenSSL dependency |
| SPAKE2 pairing | `spake2` | Implements RFC draft-irtf-cfrg-spake2, audited |
| Key generation | `ed25519-dalek` | Audited ed25519 implementation |
| Key storage encoding | `base64` | Standard encoding for key serialisation |
| Config file crypto | N/A | Keys stored at-rest unencrypted (owner-only file permissions) |

No custom cryptographic implementations. All primitives come from the above audited
crates.

## Consequences

- **`quinn` TLS configuration:** The host accepts any TLS client certificate (or none)
  since all authentication occurs over the control channel. The `ClientHello` intent
  message disambiguates between pairing and authentication flows.

- **First-run UX:** Before any client has paired, the host must print the PIN to stdout
  and wait. The CLI must handle the "no trusted clients" state gracefully.

- **Revocation ([UC-016](../uc/UC-016.md) AC #3):** Removing an entry from `trusted_clients.json` is
  sufficient. The host rejects the client's authentication attempt on the next connection.
  No CRL or OCSP needed.

- **[UC-024](../uc/UC-024.md) dependency:** The trust database path and format are part of the
  configuration persistence scope ([UC-024](../uc/UC-024.md)). The database must be readable and writable
  by the `rayplay-cli` binary without elevated permissions.

- **LAN-only assumption:** No OCSP stapling, no certificate revocation lists, no
  internet-based validation. All trust decisions are local. This is intentional and
  documented.
