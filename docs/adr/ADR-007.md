# ADR-007: Security Model (Encryption, Authentication, Trust)

- **ADR ID:** ADR-007
- **Status:** Proposed
- **Blocks:** UC-016

## Context

RayPlay streams live video, audio, and input across the network. On a shared or public
network, an unauthenticated stream is a significant security and privacy risk. The
security model must address:

1. **Authentication:** How does the host verify that a client is authorised?
2. **Encryption:** How is stream data protected from eavesdropping?
3. **Trust persistence:** How are previously-authorised clients remembered?
4. **Revocation:** How can a user remove a previously-trusted client?

Constraints:
- No PKI infrastructure (no certificate authority, no DNS)
- PIN-based pairing flow required (see UC-016)
- Must work on LAN without internet connectivity
- Cryptographic implementation should use audited, maintained Rust crates

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — QUIC + TLS 1.3 (if ADR-003 selects QUIC)**
> If the transport layer uses QUIC (`quinn`), TLS 1.3 is built in. For PIN-based pairing,
> use a PAKE (Password-Authenticated Key Exchange) protocol (e.g., SPAKE2 via the `spake2`
> crate) during the initial pairing handshake. After pairing, the client's public key is
> stored on the host as a trusted identity. Subsequent connections authenticate via the
> TLS client certificate derived from the stored key pair.
>
> **Option B — Noise Protocol Framework**
> Use the Noise Protocol Framework (`snow` crate) for mutual authentication and
> encryption. Noise `XX` handshake pattern supports mutual authentication without a CA.
> Compatible with any transport (raw UDP, TCP, or as a layer over QUIC).
>
> **Option C — Pre-shared Key (PSK) + AES-GCM**
> During PIN pairing, derive a shared secret (e.g., via HKDF from the PIN + a random
> nonce). Use this PSK for AES-256-GCM encryption of all subsequent traffic.
> Simpler than full PKI but the PSK must be re-derived if the PIN is rotated.
>
> **Recommended starting point:** Option A (QUIC + TLS 1.3 + SPAKE2) if ADR-003 selects
> QUIC — it reuses the transport's security layer rather than adding a separate one.
> If ADR-003 selects raw UDP, use Option B (Noise Protocol) as a clean, well-studied
> alternative that doesn't require QUIC.

## Consequences

- The pairing flow (UC-016) must be designed around the chosen key exchange mechanism.
  SPAKE2 and Noise both support PIN-derived authentication but with different APIs.
- Paired client identities must be stored persistently on the host (see UC-024 for
  config persistence). The storage format must protect private keys at rest.
- ADR-003 and ADR-007 are tightly coupled — the transport protocol decision constrains
  the security options. Resolve ADR-003 first, then finalise ADR-007.
- All cryptographic primitives must come from audited crates (`ring`, `rustls`, `snow`,
  `spake2`). No custom crypto implementations.
