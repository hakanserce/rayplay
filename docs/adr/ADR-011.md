# ADR-011: GUI Framework and UI Architecture

- **ADR ID:** ADR-011
- **Status:** Proposed
- **Blocks:** [Issue #101](https://github.com/hakanserce/rayplay/issues/101)
- **Depends on:** [ADR-005](ADR-005.md) (Window and Rendering Framework)

## Context

RayView is currently a CLI tool. Users must provide host addresses, ports, and
pairing PINs via command-line arguments. This works for development but is not
viable for end users. Issue #101 requests a proper GUI with host management,
PIN entry during pairing, connection status feedback, and streaming controls.

The existing rendering stack (`winit` 0.30 + `wgpu` 22, per ADR-005) already
owns the window and GPU surface. The GUI framework must layer on top of this
without breaking the zero-copy video presentation path.

## Decision

### Framework: `egui` via `egui-winit` + `egui-wgpu`

**Accepted.** `egui` 0.31 with `egui-winit` 0.31 and `egui-wgpu` 0.31.

- Immediate-mode UI — no retained widget tree, minimal state management
- `egui-winit` 0.31 supports `winit` 0.30; `egui-wgpu` 0.31 supports `wgpu` 22
- Renders as a second pass on the same swap-chain texture after the video frame,
  with alpha blending so the stream shows through transparent regions
- Lightweight — no system dependencies, pure Rust, compiles fast

**Rejected alternatives:**

- **iced** — Elm-architecture retained-mode. Heavier runtime, harder to composite
  on top of an existing wgpu render pass. Owns its own event loop, conflicting
  with the winit loop already in place.
- **Custom immediate-mode** — Too much effort for standard UI elements (text input,
  dropdowns, scrollable lists). egui provides these out of the box.

### Crate structure: `crates/rayplay-ui/`

New crate keeping UI logic separate from video rendering (SRP). Depends on
`rayplay-core` (session state) and `rayplay-network` (trust store, pairing).

The `rayplay-video` crate gains a `UiOverlay` trait and egui rendering
integration behind a `gui` feature flag. `rayplay-cli` wires them together.

### Rendering integration

```
per frame:
  1. Acquire swap-chain texture (existing)
  2. Video render pass → full-screen triangle (existing)
  3. egui render pass → alpha-blended UI on same texture (new)
  4. Submit + present
```

`wgpu_surface.rs` splits `present_to_surface` into `acquire_frame()` →
`submit_and_present()` so both passes share one surface texture.

The `UiOverlay` trait decouples `rayplay-video` from `rayplay-ui`:

```rust
pub trait UiOverlay: Send {
    fn update(&mut self, ctx: &egui::Context);
    fn wants_input(&self) -> bool;
}
```

When `wants_input()` returns `true` (UI screens visible), egui consumes
keyboard/mouse events. When `false` (streaming, no menu open), all input
passes through to the future input relay.

## UI Screens and Navigation

Interactive wireframes: [`docs/wireframes/rayview-ui.html`](../wireframes/rayview-ui.html)

### State machine

```
HostList ──[Add Host]──> AddHost ──[Save]──> HostList
   │                        │
   │                    [Cancel]──> HostList
   │
   ├──[Click Host]──> PopupMenu ──[Connect]──> Connecting (if paired)
   │                      │                        │
   │                  [Delete]──> HostList      [Connected]──> Streaming
   │                      │
   │                  [Edit]──> AddHost
   │
   │  (if unpaired)   PopupMenu ──[Connect & Pair]──> Pairing
   │                                                      │
   │                                                [Paired]──> Connecting
   │                                                [Failed]──> HostList
   │
   ├──[Settings]──> Settings ──[Back]──> HostList
   │
   └──[mDNS discovery populates list automatically]

Streaming ──[☰ fab button]──> StreamMenu
                                  ├──[Fullscreen / Exit Fullscreen]
                                  ├──[Settings]──> Settings
                                  └──[Disconnect]──> HostList

Streaming ──[connection lost]──> ReconnectBanner (top bar overlay)
                                     └──[Cancel]──> HostList
```

### Screen descriptions

- **HostList**: Grid of saved host cards showing name, IP, paired/discovered
  badges, last connected time. "+" button to add host, settings gear.
- **PopupMenu**: Context menu anchored to host card — shows host info, Connect
  (or "Connect & Pair" for unpaired hosts), Edit, Delete. No separate detail page.
- **AddHost**: Form with name, IP/hostname, port (default 7860). Save/cancel.
- **Pairing**: Shown automatically when connecting to an unpaired host. Displays
  host name, 6-digit PIN entry boxes with auto-advance, verification status,
  cancel button. On success, transitions to Connecting.
- **Connecting**: Host name, CSS spinner, cancel button.
- **Streaming**: Full-screen video. Semi-transparent ☰ floating action button
  (top-right, backdrop blur) opens a dropdown menu with: host name, stream
  stats (resolution/fps/codec/latency), connection status dot, fullscreen
  toggle, settings, disconnect.
- **ReconnectBanner**: Top bar overlay during reconnection — pulsing yellow dot,
  countdown timer, cancel button.
- **Settings**: Video section (resolution, FPS, bitrate), paired hosts with
  revoke, about/version. Back button.

### Pairing flow

Pairing is triggered implicitly on first connect to an unpaired host — the user
clicks "Connect & Pair" in the popup menu, the Pairing screen appears, and the
6-digit PIN displayed on the host must be entered. On success the host is marked
as paired and the connection proceeds. There is no separate manual "pair" action.
This replaces the current `--pair` CLI flag and `stdin` PIN prompt
(`client/connect.rs` line 205-209).

## Communication: UI ↔ Network

Bidirectional `crossbeam_channel` pair:

- `Sender<UiEvent>` held by network thread → UI (e.g. `PinRequired`,
  `Connected`, `Reconnecting`, `Disconnected`, `StreamStats`)
- `Sender<UiAction>` held by UI/render thread → network (e.g. `Connect(host)`,
  `SubmitPin(String)`, `Disconnect`)

Both channels are unbounded. The UI drains events each frame in its
`update()` call.

## Consequences

- **New dependency:** `egui` + `egui-winit` + `egui-wgpu` added to workspace.
  These are pure-Rust with no system dependencies.
- **Zero-copy path preserved:** The IOSurface video render pass is unchanged.
  egui runs as a separate pass on the same texture.
- **Feature flag:** `rayplay-video` gains egui deps behind `gui` feature to
  keep the headless/test path clean.
- **CLI backwards compatibility:** When `host` arg is provided, skip directly
  to Connecting screen. When omitted, start at HostList.
- **Coverage:** egui rendering code in `render_window.rs` is already excluded
  from coverage. `UiApp` state machine and screen logic are pure functions
  testable with headless `egui::Context`.
