# ADR-006: Input Capture Mechanism Per Platform

- **ADR ID:** ADR-006
- **Status:** Proposed
- **Blocks:** UC-009, UC-010, UC-011

## Context

Input must be captured on the client (macOS, Android) and injected on the host (Windows).
Each platform has a different input API, and "exclusive input mode" — where the client
window captures all mouse and keyboard input — requires platform-specific APIs.

Client-side (capture):
- macOS: CGEvent tap, IOHIDManager, or `winit` events
- Android: Android InputEvent API

Host-side (injection):
- Windows: `SendInput` API or `VirtualHERE`-style kernel driver

Key requirements:
- Mouse input must use relative deltas (not absolute coordinates) for gaming
- Input round-trip latency target: <2ms (see UC-009/010)
- Exclusive mode must prevent input from reaching other macOS applications

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — `winit` events (standard)**
> Use `winit`'s built-in keyboard/mouse events. Simple, cross-platform, no extra
> permissions. Con: `winit` events are window-focused — they only capture input when
> the window is focused. Exclusive mode requires additional OS-level APIs on top of
> `winit`.
>
> **Option B — CGEvent tap (macOS global hook)**
> Use the macOS CoreGraphics `CGEvent` tap to intercept all system input events globally.
> Allows capturing input even when another application is in focus. Requires
> `Accessibility` permission (user must grant in System Preferences).
> This is the only way to implement true exclusive input mode on macOS.
>
> **Option C — IOKit HID Manager**
> Read input directly from HID devices via `IOHIDManager`. Very low-level, very low
> latency. No window focus requirement. Con: complex API, requires careful device matching.
>
> **Host injection — `SendInput` (Windows)**
> The standard Windows API for synthesising keyboard and mouse events. Straightforward
> to implement. For gaming, `SendInput` with `MOUSEEVENTF_MOVE` (relative) is the
> correct approach.
>
> **Recommended approach:**
> - Client keyboard/mouse (focused window): `winit` events (Option A) for MVP
> - Client exclusive mode: `CGEvent` tap (Option B) for macOS exclusive mode (UC-011)
> - Host injection: `SendInput` (Windows) with relative mouse deltas
> - Android: standard `InputEvent` API via the Android NDK

## Consequences

- macOS `CGEvent` tap requires the `Accessibility` permission entitlement in the app
  bundle. The CLI binary must request this at launch and gracefully degrade if denied.
- `SendInput` on the host must translate between the client's logical key codes and
  Windows virtual key codes (`VK_*`). A mapping table is needed in `rayplay-input`.
- Mouse relative delta mode must be validated on macOS — `winit`'s
  `DeviceEvent::MouseMotion` provides raw deltas without cursor influence.
- The input capture path must be benchmarked end-to-end (client capture → network →
  host inject) to validate the <2ms target (see UC-009/010 ACs).
