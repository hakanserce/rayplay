# ADR-006: Input Capture Mechanism Per Platform

- **ADR ID:** ADR-006
- **Status:** Accepted
- **Blocks:** [UC-009](../uc/UC-009.md), [UC-010](../uc/UC-010.md), [UC-011](../uc/UC-011.md)

## Context

Input must be captured on the client (macOS, Android) and injected on the host (Windows).
Each platform has a different input API, and "exclusive input mode" — where the client
window captures all mouse and keyboard input — requires platform-specific APIs.

Client-side capture:
- macOS: `winit` events, CGEvent tap, IOHIDManager, or raw USB HID
- Android: Android InputEvent API, or raw USB HID via USB Host API

Host-side injection:
- Windows: `SendInput` API

Key requirements:
- Mouse input must use relative deltas (not absolute coordinates) for gaming
- Input round-trip latency target: <2ms end-to-end ([UC-009](../uc/UC-009.md)/[UC-010](../uc/UC-010.md))
- Exclusive mode must prevent input from reaching other applications ([UC-011](../uc/UC-011.md))
- No system key interception for gaming (Esc, Alt+Tab, modifier combos)

## Decision

### Option A — `winit` events (standard window input)

Use `winit`'s built-in `WindowEvent::KeyboardInput` and `DeviceEvent::MouseMotion`.
Simple, cross-platform, no extra permissions.

**Accepted for Bluetooth peripherals and fallback.** `DeviceEvent::MouseMotion`
delivers raw relative deltas. Works on both macOS and Android (via `android-activity`).
Only active while the window is focused — insufficient alone for full exclusive mode,
but sufficient as a fallback when USB is unavailable.

### Option B — CGEvent tap (macOS global hook)

Use the macOS CoreGraphics `CGEvent` tap to intercept all system input events globally.

**Accepted for macOS Bluetooth exclusive mode ([UC-011](../uc/UC-011.md)).** The only way to suppress
input from reaching the OS on macOS without raw hardware access. Requires the
`Accessibility` permission. Used only when USB HID forwarding (Option D) is unavailable
(i.e. Bluetooth peripherals).

### Option C — IOKit HID Manager (macOS raw HID reads)

Read raw HID reports from input devices via `IOHIDManager`, below the OS input stack.

**Superseded by Option D.** IOHIDManager is the correct underlying API, but Option D
packages it into a coherent cross-platform USB HID forwarding strategy. IOHIDManager
is used as the implementation detail of Option D on macOS — it is not a separate
user-visible choice.

### Option D — USB HID Forwarding (macOS + Android)

Read raw HID reports directly from USB-connected keyboards and mice using
platform-native USB/HID APIs, parse them into structured input events on the client,
and forward over the network to the host for injection via `SendInput`.

| Platform | API | Permission |
|----------|-----|------------|
| macOS | `IOHIDManager` (IOKit framework) | Input Monitoring (macOS 10.15+) |
| Android | `UsbManager` + USB Host API (API 12+) | USB permission dialog (once per device) |

**Accepted as primary path for USB-connected peripherals.** Because input is captured
below the OS input stack, the Android/macOS system never sees the events — no system
chrome interference, no key interception, no focus requirement, no Accessibility
permission on macOS.

**Scope:** Parse-and-forward, not USB/IP. The client parses raw HID reports into
structured `InputEvent` messages (key down/up with scan code, mouse delta, button
state) and sends them over the network using the existing wire protocol. The host
injects via `SendInput`. This achieves the same UX as VirtualHERE without the
overhead of forwarding raw USB packets and running a virtual USB driver on the host.

## Capture Strategy by Peripheral Type

| Peripheral | Connection | Capture mechanism |
|------------|------------|-------------------|
| Keyboard | USB | Option D (IOHIDManager / UsbManager) |
| Mouse | USB | Option D (IOHIDManager / UsbManager) |
| Keyboard | Bluetooth | Option B (CGEvent tap, macOS) / Option A (Android) |
| Mouse | Bluetooth | Option B (CGEvent tap, macOS) / Option A (Android) |

At session start, the client enumerates connected USB HID devices. If a keyboard and/or
mouse is found via USB, Option D is activated for that device. Bluetooth devices fall
back to Option B (macOS) or Option A (Android). Both paths produce the same
`InputEvent` wire format — the rest of the pipeline is unaffected.

## Host Injection: `SendInput` (Windows)

Regardless of the capture path, the host injects via `SendInput`:
- **Keyboard:** `KEYEVENTF_SCANCODE` — scan-code-based injection, reliable across
  all applications including games. A translation table in `rayplay-input` maps HID
  usage codes (from Option D) and `winit` key codes (from Options A/B) to Windows
  scan codes.
- **Mouse deltas:** `MOUSEEVENTF_MOVE` in relative mode — `dx`/`dy` carry the delta
  directly from the HID report or `DeviceEvent`.
- **Mouse buttons:** `MOUSEEVENTF_LEFTDOWN` / `MOUSEEVENTF_LEFTUP` etc. per button.

## macOS Permissions

- **Option D (IOHIDManager):** Requires Input Monitoring permission (macOS 10.15+,
  System Settings → Privacy & Security → Input Monitoring). Less intrusive than
  Accessibility — does not grant the ability to read screen content.
- **Option B (CGEvent tap):** Requires Accessibility permission.

At launch, the CLI checks which permissions are granted and activates the best
available path. Log a clear message if a permission is missing:
`"USB input capture requires Input Monitoring permission — grant it in System Settings
→ Privacy & Security → Input Monitoring for low-latency exclusive input."`

## Consequences

- **`rayplay-input` crate:** Owns the HID report parser, keycode translation table,
  `SendInput` wrapper, IOHIDManager lifecycle (macOS), USB Host lifecycle (Android),
  and CGEvent tap lifecycle (macOS fallback). Public interface: `InputCaptureSource`
  trait and `InputInjector` trait.

- **HID report parsing:** A HID usage-to-scancode translation table is needed for
  Option D. Standard HID usage tables (USB HID Usage Tables 1.3) are the reference.
  This table is shared between the macOS and Android paths.

- **Option D device enumeration:** On macOS, `IOHIDManager` is configured to match
  devices with usage page `kHIDPage_GenericDesktop` and usages `kHIDUsage_GD_Keyboard`
  / `kHIDUsage_GD_Mouse`. On Android, `UsbManager.getDeviceList()` filters by
  `UsbConstants.USB_CLASS_HID`.

- **CGEvent tap thread safety (Option B):** The tap callback runs on the main thread
  via `CFRunLoop`. It must only enqueue events into a `crossbeam::channel` — no
  blocking operations in the callback.

- **Benchmark requirement:** Before [UC-009](../uc/UC-009.md)/[UC-010](../uc/UC-010.md) are done, measure end-to-end input
  latency (client key press → network → host `SendInput` → OS acknowledgement) for
  both Option D (USB) and Option B/A (Bluetooth fallback). Both must meet the <2ms
  target on a local LAN.
