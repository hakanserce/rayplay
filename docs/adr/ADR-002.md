# ADR-002: Low-Latency Audio Capture/Encode/Send Architecture

- **ADR ID:** ADR-002
- **Status:** Accepted
- **Blocks:** UC-012, UC-013

## Context

Game audio must be captured on the Windows host, encoded, transported, and played back on
the macOS client with minimal latency. Audio/video sync drift above ~40ms is noticeable
to users. The architecture must decide on the capture API, codec, and buffering strategy.

Key constraints:
- Capture via WASAPI (Windows Audio Session API) in loopback mode
- Playback via CoreAudio on macOS
- Transport via QUIC unreliable datagrams (per ADR-003)
- Codec must minimise encode/decode latency; Opus at 10–20ms frames is the standard choice

## Decision

### Option A — WASAPI Loopback + Opus

Capture system audio via WASAPI shared loopback. Encode with Opus at **10ms frames**
(480 samples at 48 kHz, stereo). Send as QUIC unreliable datagrams. Decode on the client
with `libopus`. Play via CoreAudio with a configurable jitter buffer (default: 20ms,
range: 10–60ms).

**Accepted.** Best latency/quality/bandwidth trade-off. Opus 10ms frames give 10ms
algorithmic latency. Total path: 10ms (encode) + <1ms (transport) + 20ms (jitter buffer)
= ~31ms, well within the 40ms A/V sync tolerance. In-band FEC handles minor packet loss
without audible glitches.

### Option B — WASAPI Loopback + PCM (uncompressed)

Send raw stereo PCM (48 kHz / 16-bit, ~1.5 Mbps).

**Rejected.** Uncompressed audio provides no packet loss concealment — a single lost
datagram produces a loud glitch. Opus at 128 kbps is perceptually transparent and adds
negligible bandwidth relative to the video stream. The complexity saving is not worth the
degraded loss behaviour.

### Option C — Virtual Audio Device (VAC/VB-Cable)

Route host audio through a virtual audio device that the capture process reads.

**Rejected.** Requires a third-party driver, adds an extra hop, and introduces ~10ms of
additional latency. Not compatible with the latency budget.

## Implementation Details

### Capture: WASAPI Shared Loopback

Use WASAPI shared mode (not exclusive). Shared mode captures the system mix, which is
the correct behaviour for game audio streaming. Exclusive mode would conflict with other
audio applications and is not needed here.

WASAPI delivers audio at the system's mix format. Normalise to 48 kHz stereo before
passing to Opus. The WASAPI callback runs on a dedicated high-priority OS thread (not a
tokio task) to avoid scheduling jitter.

### Codec: Opus via `audiopus` crate

Use the `audiopus` crate (safe Rust bindings to `libopus`) for encoding on the host and
decoding on the client. Configuration:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Sample rate | 48 kHz | WASAPI and Opus native rate |
| Channels | Stereo | Game audio is stereo |
| Frame size | 480 samples (10ms) | Minimum Opus frame; halves latency vs 20ms |
| Application | `Audio` | Optimised for general audio, not voice |
| Bitrate | 128 kbps (configurable) | Perceptually transparent for stereo |
| FEC | Enabled | In-band forward error correction for packet loss |

The `audiopus` crate links against a bundled `libopus` (controlled by the `bundled`
feature flag) for hermetic builds. This FFI dependency follows the same safe-wrapper
pattern as ADR-004 — all `unsafe` confined to the crate boundary, public types are
`Send`, all errors mapped to typed `AudioError` variants.

### Transport: QUIC Unreliable Datagrams

Per ADR-003, audio frames travel on QUIC unreliable datagrams. Each datagram carries the
standard 12-byte framing header (channel = `audio`, `frag_total = 1`) followed by the
Opus payload. At 128 kbps / 10ms, each frame is ~160 bytes — well within the ~1200-byte
QUIC datagram limit. No fragmentation required.

### Jitter Buffer and Packet Loss Concealment

A short jitter buffer on the client absorbs network jitter without accumulating latency.
Default: **20ms (2 frames)**. Configurable via CLI / UC-024 in the range 10–60ms.

If a datagram is missing at its playout deadline, call `opus_decode` with a null input
to generate a PLC (packet loss concealment) frame. This produces a smooth fade rather
than a loud glitch on minor packet loss.

## Consequences

- **Dedicated WASAPI thread:** The WASAPI loopback callback runs on a dedicated
  high-priority OS thread. Encoded `AudioPacket` objects are pushed into a bounded
  `tokio::sync::mpsc` channel for the network layer to drain.

- **`audiopus` dependency:** Add to `crates/rayplay-video/Cargo.toml` with the
  `bundled` feature. Document the `libopus` build prerequisite in CLAUDE.md.

- **Configurable bitrate and jitter buffer:** Exposed as `SessionConfig` fields in
  `rayplay-core` so they can be negotiated at connect time (UC-015) and persisted
  (UC-024).

- **A/V sync:** Audio and video packets carry sender-side timestamps. The client uses
  these to align audio playback with video frame presentation. The session clock is
  established during the UC-015 capability handshake.

- **Benchmark requirement:** Before UC-012/UC-013 are done, measure round-trip audio
  latency (capture → encode → network → decode → playout) and confirm it stays within
  the 40ms A/V sync tolerance on a local LAN.
