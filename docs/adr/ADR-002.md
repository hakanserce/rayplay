# ADR-002: Low-Latency Audio Capture/Encode/Send Architecture

- **ADR ID:** ADR-002
- **Status:** Proposed
- **Blocks:** UC-012, UC-013

## Context

Game audio must be captured on the Windows host, encoded, transported, and played back on
the macOS client with minimal latency. Audio latency is perceived differently from video
latency — audio/video sync drift above ~40ms is noticeable to users. The architecture must
decide on the capture API, codec, and buffering strategy.

Key constraints:
- Capture via WASAPI (Windows Audio Session API) in loopback mode
- Playback via CoreAudio on macOS (or AudioToolbox)
- Codec must be low-latency: Opus at 20ms frames is the standard choice
- Buffer sizes must trade off latency vs. dropout risk on imperfect LANs

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — WASAPI Loopback + Opus + UDP**
> Capture system audio via WASAPI exclusive/shared loopback. Encode with Opus at 20ms
> frames (960 samples at 48 kHz). Send over the same UDP transport as video. Decode on
> client with libopus. Play via CoreAudio with a short jitter buffer (40–60ms).
>
> **Option B — WASAPI Loopback + PCM (uncompressed)**
> Skip encoding entirely. Send raw PCM over UDP. Higher bandwidth (~1.5 Mbps for stereo
> 48 kHz/16-bit) but zero encode/decode latency. Feasible on LAN.
>
> **Option C — Virtual Audio Device (VAC/VB-Cable approach)**
> Route host audio through a virtual audio device that the capture process reads.
> Adds flexibility but introduces an extra driver hop and ~10ms additional latency.
>
> **Recommended starting point:** Option A (Opus) — best latency/quality/bandwidth
> trade-off. 20ms Opus frames + 40ms jitter buffer = ~60ms audio latency, which is
> within acceptable A/V sync tolerance. Option B is acceptable for initial MVP if Opus
> FFI proves complex.

## Consequences

- Opus requires a dependency on `libopus` (C library) via FFI or the `audiopus` crate.
  This must be vetted against the project's FFI approach (see ADR-004).
- WASAPI exclusive mode gives lowest capture latency but may conflict with other audio
  applications. Shared mode is safer for initial implementation.
- A jitter buffer is required on the client; its size is a runtime trade-off between
  latency and dropout resistance — make it configurable (see UC-024).
