# ADR-005: Window and Rendering Framework

- **ADR ID:** ADR-005
- **Status:** Proposed
- **Blocks:** UC-005, UC-022

## Context

The RayView client must create a window, receive decoded video frames, and render them
as fast as possible. The rendering framework must support:

- macOS as the primary platform
- Android as a planned future platform (UC-022)
- Presenting decoded frames (from VideoToolbox / MediaCodec) with minimal copy/latency
- Exclusive input capture (the window must intercept all keyboard/mouse input — see UC-011)
- Sub-frame presentation (frames should be displayed as soon as decoded, not queued)

## Decision

> **To be decided.** Options under consideration:
>
> **Option A — `winit` + `wgpu`**
> `winit` for window creation and event loop (cross-platform: macOS, Windows, Android,
> web). `wgpu` for GPU rendering. Decoded frames uploaded as textures and rendered via
> a full-screen quad shader.
> Pro: cross-platform, idiomatic Rust, active ecosystem. Con: `wgpu` texture upload adds
> one copy if frames are not in a `wgpu`-compatible format natively.
>
> **Option B — `winit` + Metal (macOS) / Vulkan (Android)**
> `winit` for windowing, platform-native graphics APIs for rendering. Use `metal-rs` on
> macOS, `ash` (Vulkan) on Android. Allows zero-copy presentation if decoded frames are
> in a Metal/Vulkan-compatible format.
> Pro: optimal performance. Con: platform-specific rendering code.
>
> **Option C — `sdl2`**
> SDL2 provides windowing, input, and rendering in one library. Well-tested for games.
> Con: C dependency, less idiomatic Rust, SDL2's input handling may conflict with
> exclusive input requirements.
>
> **Recommended starting point:** Option A (`winit` + `wgpu`) — best balance of
> cross-platform support and Rust ecosystem fit. If the texture upload copy proves to
> exceed the latency budget, revisit Option B for macOS. The Android client (UC-022) will
> benefit from `winit`'s Android support via `android-activity`.

## Consequences

- `winit` is the window/event abstraction; its event loop must be integrated with the
  `tokio` async runtime (use `winit`'s `EventLoopProxy` + a dedicated render thread).
- `wgpu` requires a one-time setup of device, queue, swap chain, and shaders. This code
  lives in `rayplay-video` (rendering layer) with a clear interface boundary.
- Exclusive input mode (UC-011) interacts with `winit`'s cursor grab API
  (`window.set_cursor_grab(CursorGrabMode::Locked)`). Validate this works on macOS before
  committing to Option A.
- For Android (UC-022), `winit` supports Android via the `android-activity` crate.
  The rendering layer must be kept sufficiently abstract to support platform-specific
  surface types.
