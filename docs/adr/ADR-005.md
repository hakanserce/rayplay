# ADR-005: Window and Rendering Framework

- **ADR ID:** ADR-005
- **Status:** Accepted
- **Blocks:** [UC-005](../uc/UC-005.md), [UC-022](../uc/UC-022.md)

## Context

The RayView client must create a window, receive decoded video frames, and render them
as fast as possible. The rendering framework must support:

- macOS as the primary platform (Apple Silicon)
- Android as a planned future platform ([UC-022](../uc/UC-022.md))
- Presenting decoded frames from VideoToolbox (macOS) / MediaCodec (Android) with
  minimal copy and latency
- Exclusive input capture — the window must intercept all keyboard/mouse input ([UC-011](../uc/UC-011.md))
- Sub-frame presentation — frames must be displayed as soon as decoded, not queued

## Decision

### Option A — `winit` + `wgpu`

`winit` for window creation and event loop (cross-platform: macOS, Windows, Android).
`wgpu` for GPU rendering via a full-screen quad shader. Decoded frames uploaded as
textures and rendered to the swap chain.

**Accepted.** Cross-platform, idiomatic Rust, active ecosystem. `wgpu` on macOS uses
the Metal backend, which supports zero-copy texture import from a VideoToolbox
`CVPixelBuffer` via IOSurface (see below). Android support via `winit`'s
`android-activity` crate aligns with [UC-022](../uc/UC-022.md).

### Option B — `winit` + Metal (macOS) / Vulkan (Android)

`winit` for windowing, platform-native graphics APIs (`metal-rs` / `ash`) for rendering.

**Rejected as primary** but retained as an escape hatch. If `wgpu`'s Metal backend does
not support IOSurface external texture import in a future version, drop `wgpu` and use
`metal-rs` directly for the macOS render path. The `Renderer` trait boundary (see below)
makes this swap localised to `rayplay-video`.

### Option C — `sdl2`

SDL2 for windowing, input, and rendering.

**Rejected.** C dependency, less idiomatic Rust. SDL2's input handling conflicts with
the exclusive input mode requirements ([UC-011](../uc/UC-011.md)) and would require bypassing SDL2's event
loop entirely for CGEvent tap integration ([ADR-006](ADR-006.md)).

---

## Zero-Copy Frame Presentation (macOS)

VideoToolbox decodes into a `CVPixelBuffer` backed by an `IOSurface`. On the Metal
backend, `wgpu` can import an IOSurface as an external texture via
`wgpu::Device::create_texture_from_iosurface` (exposed through the `wgpu` Metal
interop extension). This avoids a GPU→CPU→GPU round-trip.

The presentation path is:
```
VideoToolbox decode → CVPixelBuffer (IOSurface) → wgpu external texture → full-screen quad
```

If the IOSurface import API is unavailable for a given `wgpu` version, fall back to
uploading pixel data via `wgpu::Queue::write_texture`. This adds one copy but remains
within the <2ms rendering budget at 1080p.

## Event Loop and Thread Model

`winit`'s event loop runs on the main thread (required by macOS AppKit). The render
thread is the same thread. Decoded frames are delivered from the decode thread via a
bounded `crossbeam::channel` (not `tokio::sync` — the render loop is synchronous).

The event loop calls `window.request_redraw()` on each new frame arrival and renders
in the `RedrawRequested` handler. This ensures frames are presented at the display's
vsync cadence without an extra buffer hop.

## Exclusive Input Mode

`winit`'s `window.set_cursor_grab(CursorGrabMode::Locked)` locks the cursor on macOS.
Validate this works on macOS 13+ before UC-011 is considered done — cursor lock
behaviour changed in macOS 13 and requires `NSWindow` to be key and front.

For the CGEvent tap ([ADR-006](ADR-006.md) Option B), the tap is installed alongside the `winit`
event loop on the main thread. Events captured by the tap are filtered before they
reach `winit`, so there is no double-delivery.

## Android ([UC-022](../uc/UC-022.md))

`winit` supports Android via the `android-activity` crate. The render surface is an
`ANativeWindow` presented as a `wgpu::Surface`. MediaCodec decodes into a
`SurfaceTexture`, which can be imported into `wgpu` via EGL external texture on the
Vulkan backend. The same `Renderer` trait is implemented for Android — no change to the
rendering pipeline logic.

## Consequences

- **`Renderer` trait in `rayplay-video`:** A thin `Renderer` trait with a single
  `present_frame(&mut self, frame: &DecodedFrame)` method isolates platform-specific
  rendering code. `WgpuRenderer` implements it for macOS/Android; a hypothetical
  `MetalRenderer` would implement it for the Option B escape hatch.

- **`winit` event loop integration:** `winit` owns the main thread. The CLI entry point
  must hand control to `winit`'s event loop after spawning all media threads. Use
  `EventLoopBuilder` to pass a channel sender into the loop closure.

- **Swap chain configuration:** Present mode `Mailbox` (if available) or `Fifo`.
  `Mailbox` drops frames that are not yet presented when a new one arrives — correct
  behaviour for a streaming client where freshness beats completeness.

- **Benchmark requirement:** Before [UC-005](../uc/UC-005.md) is done, confirm render-to-present latency
  is <2ms at 1080p60 on Apple Silicon with the IOSurface import path active.
